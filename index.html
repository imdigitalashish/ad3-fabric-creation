Service Level Indicators

Why we create SLIs

It's 2 AM. Dashboards green, CPU fine, server up — yet users are angry. "Up" isn't "working well."

  • Measure what users feel — facts, not guesses
  • End the "is it broken?" debate — above the line, or below it
  • Spend the error budget wisely — SLIs power SLOs
  • Alert on pain, not noise — page humans when users hurt

Measure the user's happiness — not the machine's vital signs.

How it started

  • Before authoring SLIs, we first needed to know what to measure and which objectives to target.
  • So we began by analyzing a lot of incidents.
  • It was all manual — so I built ZPI to capture what everyone was doing into memory, making the whole process far faster.
  • With every incident mapped to a CUJ, we identified the top contributing factors and started picking them off.
