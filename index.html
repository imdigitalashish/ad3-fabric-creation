02

Authoring the SLIs

Phase 01 — Signal

Telemetry analysis

Once we know the failing component, we ask: what telemetry does it already emit?

  • Telemetry is the user's voice — it tells us what they actually feel
  • Find the signal that maps to what we want to measure
Phase 02 — Measure

Authoring the KQL

Two ingredients, one query: what to measure × how to measure it.

  • Step 1 gave the signal; step 2 gave the telemetry
  • Write the KQL that turns them into a ratio
  • Then prove it actually moves when things break
Phase 03 — Prove

Backtesting

A simple idea: replay the past and see if the SLI would have noticed.

  • Spin up a simulation environment
  • Fire the KQL exactly as it would run in production
  • Watch for the reliability dip during known incidents
  • Catches the trap of measuring the wrong signal
Phase 04 — Emit

Flow mapping

Emission is easy. Knowing where to emit is the hard part.

  • Emit total and valid events to Geneva, tagged by dimension
  • The catch: where in the code does the path actually fail?
  • An LLM alone hallucinates the critical path
  • So I used Roslyn — parse to an AST, then trace the real path
